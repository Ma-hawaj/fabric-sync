This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Fabric Sync is an early-stage full-stack app for a tailoring/garment business (customer records, body measurements, orders/invoices — see the mock data in `frontend/src/features/customers` and `orders` for domain vocabulary like thobe types, collar/cuff styles, etc.). It is split into:

- `backend/` — Rust/Axum HTTP API
- `frontend/` — React/TanStack Router SPA

**The codebase is currently a minimal skeleton, well ahead of the surrounding docs.** `README.md` describes a target architecture (Postgres via SQLx, OAuth2/OIDC via Zitadel, real login) that is only partially wired up:

- The backend has no database connection, no SQLx dependency, and no `migrations/` directory yet. The `users` feature's repository layer (`backend/src/features/users/repository.rs`) just returns an empty `Vec`.
- There is no `auth.rs` / token-introspection middleware in the backend yet.
- The frontend's `AuthProvider` (`frontend/src/lib/auth.tsx`) hardcodes `isAuthenticated: true` — there is no real login screen or token storage yet, and the `/login` route referenced by the `_authenticated` layout's redirect doesn't exist (the redirect target is currently commented out and falls back to `/`).
- `docker-compose.yml` stands up Zitadel + Postgres, but the backend doesn't read `DATABASE_URL`/OAuth env vars yet — `config.rs` only reads `PORT`.
- Frontend feature hooks (`use-customers.ts`, `use-orders.ts`) return hardcoded mock arrays behind a simulated network delay, not real API calls. `lib/api.ts` only exports `apiBaseUrl`; there's no fetch wrapper yet.

When asked to add backend persistence, auth, or wire the frontend to real endpoints, treat this as new work, not "restoring" something — check the actual files rather than assuming the README's description already holds.

## Commands

### Backend (`cd backend`)

```bash
cargo run              # start the API (reads PORT env var, default 3000)
cargo check             # type-check
cargo fmt --check       # verify formatting (CI-equivalent check)
cargo fmt               # apply formatting
```

There is no backend test suite yet.

### Frontend (`cd frontend`)

The frontend uses **pnpm** (`pnpm-lock.yaml` + `pnpm-workspace.yaml` are checked in — there is no `package-lock.json`), even though the root `README.md` and the boilerplate `frontend/README.md` show `npm` commands.

```bash
pnpm install
pnpm dev                # vite dev server on port 3000
pnpm build               # production build
pnpm test                # vitest run (no test files exist yet)
pnpm lint                # eslint
pnpm format               # prettier --write . && eslint --fix
pnpm check                # prettier --check . (CI-equivalent check)
pnpm generate-routes       # regenerate src/routeTree.gen.ts from src/routes
```

`src/routeTree.gen.ts` is generated by the TanStack Router Vite plugin (also runs automatically on `pnpm dev`/`pnpm build`) — never hand-edit it.

To point the dev server at a backend running on a different origin/port:

```bash
VITE_API_BASE_URL=http://localhost:3001 pnpm dev
```

### Local infrastructure

```bash
cp .env.example .env
docker compose up -d --wait
```

Brings up a local Zitadel (OIDC issuer, console at `http://localhost:8080/ui/console`) and Postgres (`postgres://postgres:postgres@localhost:5432/fabric_sync`), adapted from Zitadel's official compose reference. Not currently consumed by the backend (see above).

## Backend architecture

`main.rs` wires together tracing, `Config::from_env()`, `AppState`, and `app::router()`, then serves with `axum::serve`.

- **`app.rs`** — composes the top-level `Router` by merging `features::router()` and attaching `TraceLayer`.
- **`state.rs`** — `AppState` is the single `Clone`-able struct injected via `.with_state()`; add new shared resources (DB pool, HTTP clients) here.
- **`config.rs`** — env-var-backed `Config`, constructed once in `main`.
- **`error.rs`** — `AppError` enum implementing `IntoResponse` (and `From` for wrapped error types) so handlers can return `Result<_, AppError>`; add a new variant + `From` impl per new failure mode rather than stringly-typed errors.
- **`features/`** — one module per vertical slice, merged in `features/mod.rs::router()`. Each feature follows a layered structure (see `features/users/` as the template):
  - `routes.rs` — builds the feature's `Router<AppState>` and maps paths to handlers
  - `handlers.rs` — axum handler functions (extract state/params, call `service`, wrap in `Json`)
  - `service.rs` — business logic, operates on `AppState` and calls `repository`
  - `repository.rs` — data access (currently in-memory/stubbed; this is where DB queries will go)
  - `types.rs` — domain types plus their `...Response` DTOs and `From` conversions
  - `mod.rs` — keeps submodules private and re-exports only `router`

  New features should follow this same `routes -> handlers -> service -> repository -> types` split and register their router in `features/mod.rs`.

## Frontend architecture

- **Routing**: TanStack Router, file-based, under `src/routes/`. `_authenticated.tsx` is a pathless layout route gating everything under `_authenticated/` via `beforeLoad` checking `context.auth.isAuthenticated` (router context is assembled in `main.tsx` from `useAuth()` + a shared `QueryClient`).
- **Data fetching**: TanStack Query. Each feature owns its query hook(s) under `features/<feature>/hooks/`. Currently these hooks return mock data (with an artificial delay) rather than hitting the backend.
- **Feature module convention** (`src/features/<feature>/`): `components/`, `hooks/`, `types/`, plus a top-level `<feature>.tsx` page component composing them. Follow this layout for new features (see `customers/` as the fullest example; `users` is only wired at the route level with an inline `<table>` and hasn't adopted this structure yet).
- **Data table stack**: `components/data-table/*` + `hooks/use-data-table.ts` + `lib/data-table.ts` + `lib/parsers.ts` + `config/data-table.ts` + `types/data-table.ts` together form a reusable advanced table (TanStack Table + `nuqs` for URL-synced pagination/sorting/filtering state, in the shadcn/diceui data-table style). Use this stack for new listing pages (see `customers/customers.tsx`) rather than hand-rolling a table.
- **UI primitives**: `components/ui/*` are shadcn-style/Base UI primitives, configured via `components.json` (style `base-rhea`, neutral base color, **`rtl: true`** — keep RTL layouts in mind for new UI). Icons via `lucide-react`.
- **Path aliases**: both `@/*` and `#/*` resolve to `src/*` (see `tsconfig.json` paths and `package.json` `imports`) — existing files use either interchangeably.
- **Styling**: Tailwind CSS 4 via `@tailwindcss/vite`, theme/globals in `src/styles.css`.
